# Protocol

This document describes the current encrypted transport protocol implemented in `0.5.0`.

It is intentionally small and versioned so that the reference relay server can remain replaceable and the same message contract can be reused by other transports.

## Scope

The current protocol definition applies to relay transport and the direct IPv6 phone route.

The local-only route still needs to adopt the same encrypted message contract.

## Pairing Link

Direct-IPv6 pairing links carry their session metadata in the URL fragment:

```text
#mode=<route>&s=<session-id>&k=<pairing-secret>
```

- `s` is the session identifier.
- `k` is a base64url pairing secret generated by LibreOffice.
- `mode` is `ipv6`.

Relay pairing links add a relay admission token:

```text
#mode=relay&s=<session-id>&k=<pairing-secret>&a=<admission-token>
```

- `a` is the relay admission token required for `/api/session` and `/ws`.

The fragment is not sent in HTTP requests to the relay server, but it is visible to the JavaScript running in the loaded page.

## Message Types

All encrypted transport messages are UTF-8 JSON objects.

### `hello`

The plugin sends plaintext `hello` messages to advertise the active key material.

Example:

```json
{
  "type": "hello",
  "v": 1,
  "s": "demo-session",
  "k": "kid123",
  "nonce": "pluginnonce",
  "suite": "HKDF-SHA256+AES-256-GCM+PSK",
  "rotate": 300,
  "features": ["state", "command", "error"]
}
```

Fields:

- `v`: protocol version
- `s`: session id
- `k`: key id for this key generation
- `nonce`: base64url plugin nonce used during HKDF derivation
- `suite`: cipher-suite label
- `rotate`: maximum number of messages before the plugin rotates its send key
- `features`: currently supported encrypted frame kinds

### `frame`

Encrypted transport traffic uses `frame` messages.

Example:

```json
{
  "type": "frame",
  "v": 1,
  "s": "demo-session",
  "k": "kid123",
  "kind": "state",
  "n": "framenonce",
  "ct": "ciphertext-and-tag"
}
```

Fields:

- `v`: protocol version
- `s`: session id
- `k`: key id from the most recent accepted `hello`
- `kind`: `state`, `command`, `error`, or `asset`
- `n`: base64url 96-bit AES-GCM nonce
- `ct`: base64url ciphertext with the 128-bit AES-GCM tag appended

Directions:

- plugin -> phone: `state`, `error`, `asset`
- phone -> plugin: `command`

### `error`

Plaintext `error` messages are available for transport-level failures before a secure key is active.

Example:

```json
{
  "type": "error",
  "v": 1,
  "s": "demo-session",
  "code": "unsupported-version",
  "message": "Unsupported relay protocol version."
}
```

## Key Derivation

The current `0.5.0` bootstrap is pre-shared-secret based.

Inputs:

- pairing secret: base64url-decoded `k`
- session id: `s`
- key id: `k` from the `hello` message
- plugin nonce: `nonce` from the `hello` message

Derivation:

```text
salt = "impress-remote-relay/v1" || 0x00 || session-id
info = "relay-keys" || 0x00 || key-id || 0x00 || plugin-nonce
material = HKDF-SHA256(pairing-secret, salt, info, 64 bytes)
state-key = material[0:32]
command-key = material[32:64]
```

## Encryption Rules

- `state-key` encrypts plugin-to-phone `state`, `error`, and `asset` frames.
- `command-key` encrypts phone-to-plugin `command` frames.
- AES-GCM uses a 96-bit random nonce and a 128-bit authentication tag.
- Additional authenticated data is the compact JSON object:

```json
{"kind":"state","k":"kid123","n":"framenonce","s":"demo-session","v":1}
```

That binds every encrypted frame to the expected session, key id, nonce, frame kind, and protocol version.

## Replay Protection and Rotation

- Both sides keep recent nonce caches and reject replayed encrypted frames.
- The plugin rotates its send key after the configured message threshold and emits a fresh `hello`.
- The plugin keeps a small window of older receive keys so in-flight phone commands can survive key rotation.

## Relay Server Behavior

The relay server:

- matches websocket peers by session id
- requires the relay admission token on `/ws` and `/api/session`
- forwards plaintext `hello`
- forwards opaque encrypted `frame` messages
- may replay the latest `hello` and a bounded window of recent encrypted plugin frames to a newly joined phone while the plugin remains connected
- exposes `/api/session` so LibreOffice can detect joined phones before the first command
- rate-limits noisy websocket peers and bounds replay caches
- clears cached secure state when the plugin disconnects

The relay server must not decrypt frames, inspect presenter notes, or persist slide previews. It should validate only the outer envelope enough to reject malformed or legacy traffic and then forward or replay opaque ciphertext.

## Direct IPv6 HTTP Behavior

The direct IPv6 route uses the same `hello` and `frame` payloads over the LibreOffice-served HTTP app:

- `/api/direct/handshake` returns the current plaintext `hello`
- `/api/direct/state` returns an encrypted `state` frame, plus a fresh `hello` when the send key rotates
- `/api/direct/events` streams `hello` and encrypted `state` events over SSE
- `/api/direct/command` accepts encrypted `command` frames from the phone
- `/api/direct/slide/current` and `/api/direct/slide/next` return encrypted `asset` frames that contain PNG slide data

## Current Limitations

- Relay bootstrap still uses a shared pairing secret rather than ECDH P-256.
- The local-only route does not yet use this encrypted frame protocol.
- A malicious relay-hosted phone UI can still serve hostile JavaScript unless the frontend is pinned or delivered through a trusted path.
