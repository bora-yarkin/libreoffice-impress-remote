# Protocol

This document describes the current encrypted transport protocol implemented in `1.0.6`.

It is intentionally small and versioned so that the reference relay server can remain replaceable and the same message contract can be reused by other transports.

## Scope

The current encrypted protocol definition applies to relay, direct IPv6, and local transport when the browser exposes Web Crypto. Local mode also has a documented authenticated plaintext compatibility fallback described later in this document.

## Pairing Link

Local, LocalTunnel, and direct-IPv6 pairing links carry their session metadata in the URL fragment. Users normally receive this link by scanning the QR code or using `Copy URL` in the QR popup.

```text
#mode=<route>&s=<session-id>&k=<pairing-verifier>
```

- `s` is the session identifier.
- `k` is a base64url pairing verifier generated by LibreOffice. It is mixed into ECDH key derivation and is also used by Safari-style HTTP compatibility fallback.
- `mode` is `local`, `tunnel`, or `ipv6`.

Relay pairing links add a relay admission token:

```text
#mode=relay&s=<session-id>&k=<pairing-verifier>&a=<admission-token>
```

- `a` is the relay admission token required for `/api/session` and `/ws`.

The fragment is not sent in HTTP requests to the relay server, but it is visible to the JavaScript running in the loaded page.

## Message Types

All encrypted transport messages are UTF-8 JSON objects.

### `hello`

Encrypted transports use plaintext `hello` messages for ECDH negotiation. The plugin sends a plugin hello first; the phone answers with a phone hello for the same key id and plugin nonce.

Plugin hello example:

```json
{
  "type": "hello",
  "v": 1,
  "s": "demo-session",
  "k": "kid123",
  "nonce": "pluginnonce",
  "role": "plugin",
  "pub": "base64url-p256-plugin-public-key",
  "suite": "ECDH-P256+HKDF-SHA256+AES-256-GCM",
  "rotate": 300,
  "features": ["state", "command", "error", "asset"]
}
```

Phone hello example:

```json
{
  "type": "hello",
  "v": 1,
  "s": "demo-session",
  "k": "kid123",
  "nonce": "pluginnonce",
  "role": "phone",
  "pub": "base64url-p256-phone-public-key",
  "suite": "ECDH-P256+HKDF-SHA256+AES-256-GCM",
  "rotate": 300,
  "features": ["state", "command", "error", "asset"]
}
```

Fields:

- `v`: protocol version
- `s`: session id
- `k`: key id for this key generation
- `nonce`: base64url plugin nonce used during HKDF derivation
- `role`: `plugin` or `phone`
- `pub`: base64url raw uncompressed P-256 public key
- `suite`: cipher-suite label
- `rotate`: maximum number of messages before the plugin rotates its send key
- `features`: currently supported encrypted frame kinds

### `frame`

Encrypted transport traffic uses `frame` messages.

Example:

```json
{
  "type": "frame",
  "v": 1,
  "s": "demo-session",
  "k": "kid123",
  "kind": "state",
  "n": "framenonce",
  "ct": "ciphertext-and-tag"
}
```

Fields:

- `v`: protocol version
- `s`: session id
- `k`: key id from the most recent accepted `hello`
- `kind`: `state`, `command`, `error`, or `asset`
- `n`: base64url 96-bit AES-GCM nonce
- `ct`: base64url ciphertext with the 128-bit AES-GCM tag appended

Directions:

- plugin -> phone: `state`, `error`, `asset`
- phone -> plugin: `command`

### `error`

Plaintext `error` messages are available for transport-level failures before a secure key is active.

Example:

```json
{
  "type": "error",
  "v": 1,
  "s": "demo-session",
  "code": "unsupported-version",
  "message": "Unsupported relay protocol version."
}
```

## Key Derivation

The current `1.0.6` bootstrap uses ephemeral P-256 ECDH.

Inputs:

- ECDH shared secret from plugin private key and phone public key, or phone private key and plugin public key
- pairing verifier: base64url-decoded `k` from the pairing link
- session id: `s`
- key id: `k` from the `hello` message
- plugin nonce: `nonce` from the `hello` message
- plugin public key: `pub` from the plugin hello
- phone public key: `pub` from the phone hello

Derivation:

```text
salt = "impress-remote-relay/v1" || 0x00 || session-id || 0x00 || pairing-verifier
info = "relay-keys" || 0x00 || key-id || 0x00 || plugin-nonce || 0x00 || plugin-public-key || 0x00 || phone-public-key
material = HKDF-SHA256(ecdh-shared-secret, salt, info, 64 bytes)
state-key = material[0:32]
command-key = material[32:64]
```

## Encryption Rules

- `state-key` encrypts plugin-to-phone `state`, `error`, and `asset` frames.
- `command-key` encrypts phone-to-plugin `command` frames.
- AES-GCM uses a 96-bit random nonce and a 128-bit authentication tag.
- Additional authenticated data is the compact JSON object:

```json
{"kind":"state","k":"kid123","n":"framenonce","s":"demo-session","v":1}
```

That binds every encrypted frame to the expected session, key id, nonce, frame kind, and protocol version.

## Replay Protection and Rotation

- Both sides keep recent nonce caches and reject replayed encrypted frames.
- The plugin rotates its send key after the configured message threshold and emits a fresh `hello`.
- The plugin keeps a small window of older receive keys so in-flight phone commands can survive key rotation.

## Relay Server Behavior

The relay server:

- matches websocket peers by session id
- requires the relay admission token on `/ws` and `/api/session`
- forwards plaintext plugin and phone `hello` messages
- forwards opaque encrypted `frame` messages
- may replay the latest `hello` and a bounded window of recent encrypted plugin frames to a newly joined phone while the plugin remains connected
- exposes `/api/session` so LibreOffice can detect joined phones before the first command
- rate-limits noisy websocket peers and bounds replay caches
- clears cached secure state when the plugin disconnects

The relay server must not decrypt frames, inspect presenter notes, or persist slide previews. It should validate only the outer envelope enough to reject malformed or legacy traffic and then forward or replay opaque ciphertext.

## Local and Direct IPv6 HTTP Behavior

Local, LocalTunnel, and direct IPv6 routes use the same `hello` and `frame` payloads over the LibreOffice-served HTTP app:

- `/api/direct/handshake?s=<session-id>` returns the current plaintext `hello`
- `POST /api/direct/handshake?s=<session-id>` accepts the phone `hello` response and completes ECDH
- `/api/direct/state?s=<session-id>` returns an encrypted `state` frame, plus a fresh `hello` when the send key rotates
- `/api/direct/events?s=<session-id>` streams `hello` and encrypted `state` events over SSE
- `/api/direct/command?s=<session-id>` accepts encrypted `command` frames from the phone
- `/api/direct/slide/current?s=<session-id>&rev=<revision>` and `/api/direct/slide/next?s=<session-id>&rev=<revision>` return encrypted `asset` frames that contain PNG slide data

The embedded server intentionally does not expose the old unauthenticated plaintext state, command, event, or slide-image endpoints. Direct encrypted endpoints require the session id from the QR/Copy URL fragment before the request is treated as paired client activity. This does not replace encryption or reveal the pairing verifier, but it prevents unrelated LAN probes from triggering connected-client behavior.

Local mode has one compatibility exception for Safari and other browsers that do not expose Web Crypto on plain LAN HTTP origins:

- `/api/local/state` returns plaintext state and requires `X-Impress-Remote-Session` plus `X-Impress-Remote-Secret`
- `/api/local/command` accepts plaintext commands and requires the same headers
- `/api/local/slide/current` and `/api/local/slide/next` return PNG slide data and require the same headers
- no plaintext event stream is exposed because `EventSource` cannot attach the pairing verifier header
- `/api/local/*` is restricted to loopback, link-local, or private/local-network client addresses

The phone UI selects this fallback only for `mode=local` when Web Crypto is missing. Direct IPv6 and relay mode remain encrypted-only and fail closed without Web Crypto.

Slide asset endpoints reject stale render revisions with `409 Conflict` so a client does not silently receive an image for a different current or next slide than the state frame advertised.

The local and direct IPv6 routes still serve the static phone web shell over HTTP, so encrypted frames protect presenter data and commands from passive observers when Web Crypto is available, but local HTTPS or frontend pinning would still be needed to resist an active attacker that can modify the JavaScript before it runs. The Safari-style compatibility fallback is authenticated but plaintext. Local mode restricts that fallback to local-network clients; direct IPv6 mode permits it only when the user explicitly chose the experimental IPv6 route.

All embedded local responses include no-store caching plus browser hardening headers such as `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`, frame denial, and a restrictive Content Security Policy.

## Current Limitations

- A malicious relay-hosted phone UI can still serve hostile JavaScript unless the frontend is pinned or delivered through a trusted path.
- The local route does not currently use HTTPS for the static phone web shell.
- Safari-style HTTP compatibility is not E2EE because Safari does not expose Web Crypto on the affected plain HTTP origin.
