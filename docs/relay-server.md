# Relay Server

The relay server is the optional fallback path for networks where local mode and direct IPv6 do not work.

It is designed to stay thin:

- no database
- no server-side LibreOffice install
- no presenter-note decryption
- no slide-preview parsing
- no Node.js requirement for the Python relay

The repository ships two compatible relay deployments:

- a standalone Python relay bundle for VPSes, home servers, and ordinary Linux or Windows hosts
- a Cloudflare Worker plus Durable Object bundle for a free-tier-compatible serverless deployment

Both serve the same shared phone UI from `shared/webui/`.

## What The Relay Serves

The reference relay exposes:

- `/` plus `/app.js` and `/app.css` for the lightweight phone remote
- `/asset-manifest.json` so published web assets can be verified
- `/health` for runtime and limit information
- `/api/session?session=...&a=...` for admission-controlled session status
- `/ws?role=plugin|phone&session=...&a=...` for the relay websocket transport

`a` is the relay admission token generated by LibreOffice and embedded in the QR/manual link fragment for relay mode.

## Security Model

Relay mode currently has two separate trust boundaries:

1. Transport confidentiality
2. Frontend delivery trust

Transport confidentiality is already implemented:

- LibreOffice generates the pairing secret and relay admission token
- relay `frame` payloads stay AES-GCM encrypted end to end
- the relay only sees session metadata, plaintext `hello` metadata, and opaque ciphertext frames

Frontend delivery trust is intentionally simpler in `0.6.12`:

- the supported production model is self-hosting the published relay bundle
- the bundle ships `asset-manifest.json` with SHA-256 hashes of the served UI files
- users or packagers can compare that manifest with the release artifact before trusting a deployment

That means a third-party relay can still serve hostile JavaScript if you choose to trust its hosted UI. The current release is safe against an honest-but-curious relay, not against a relay operator you do not trust to serve the frontend.

## Relay Pairing Flow

The relay phone link currently looks like:

```text
https://relay.example.com/#mode=relay&s=<session>&k=<pairing-secret>&a=<admission-token>
```

- `s` identifies the relay session
- `k` is the pairing secret used for encrypted relay keys
- `a` is the admission token required for `/api/session` and `/ws`

The fragment stays out of normal HTTP requests, but the JavaScript loaded from that page can read it.

## Runtime Behavior

The reference relay now supports:

- admission-controlled session creation and join
- plugin replacement when a newer LibreOffice runtime reconnects to the same session
- replay of the latest `hello` plus a bounded cache of encrypted plugin frames to newly joined phones
- session-status polling so LibreOffice can detect that a relay phone joined before the first button press
- automatic cleanup when a session becomes empty
- rate limiting, phone-count caps, message-size caps, and session-id validation
- structured logs and per-session metrics for the reference runtimes

## Python Relay Bundle

Build it from the repo root with:

```bash
make release-bundle
```

For a source checkout development run:

```bash
make server-dev
```

For the stripped release bundle:

```bash
make release-bundle
cd dist/impress-remote-relay-python-0.6.12
./run-relay.sh
```

The installed `.oxt` also contains the matching Python relay bundle, Cloudflare relay bundle, and documentation bundle for its own version. In LibreOffice Impress, open `Slide Show -> Presentation Remote -> Advanced Remote Settings`, choose `Get Relay Server`, `Get Cloudflare Relay`, or `Get Documentation`, and export it to Downloads or a selected folder.

Windows PowerShell:

```powershell
.\run-relay.ps1
```

The standalone bundle also includes:

- `install-linux-service.sh`
- `uninstall-linux-service.sh`
- `install-windows-service.ps1`
- `uninstall-windows-service.ps1`

Those helpers persist a randomly chosen free port in `data/service.json`.

## Cloudflare Bundle

Build it from the repo root with:

```bash
make cloudflare-bundle
```

Then deploy with Wrangler from the unpacked bundle directory:

```bash
npx wrangler deploy
```

The Worker serves static UI assets from `public/` and keeps relay session state in Durable Objects.

## Reverse Proxy And TLS Notes

If you deploy the Python relay behind Nginx, Caddy, Traefik, or another reverse proxy:

- terminate TLS there or run plain HTTP only on a trusted internal network
- forward websocket upgrades on `/ws`
- preserve query parameters on `/ws` and `/api/session`
- do not strip the `Upgrade` and `Connection` headers
- keep `/asset-manifest.json` reachable if you want remote asset verification

The relay can live on a subpath, but the proxy must keep `/ws`, `/api/session`, and the static UI paths aligned with the configured relay base URL.

## Firewall Notes

If you expose the relay directly:

- allow inbound TCP only on the chosen relay port
- prefer HTTPS or TLS-terminated websocket access on the public internet
- keep the relay behind ordinary host firewall rules
- treat the relay admission token as required session authorization, not as a substitute for host-level firewalling

## Basic Verification

Health:

```bash
curl http://relay-host:17865/health
```

Session status:

```bash
curl 'http://relay-host:17865/api/session?session=<session-id>&a=<admission-token>'
```

Asset-manifest verification:

```bash
curl http://relay-host:17865/asset-manifest.json
```

LibreOffice relay pairing:

1. Configure the relay base URL in LibreOffice.
2. Start the remote from Impress.
3. Scan the QR code or open the full manual relay link with its `#...` fragment intact.
4. LibreOffice should detect the joined phone and start the slideshow automatically.
